Test Mobile App Changes
Use Case Model

Scope: Once the Mobile App Lead receives the updated mobile application they distribute the MobileAppTestPlan to Testers. Each Tester completes their assigned test and sends their results to the Mobile App Lead, along with any MobileAppIssues they encountered. The Mobile App Lead collects all test results, creates a IssueLog for the MobileAppIssues, and sends Notification to stakeholders on the testing progress. The Mobile App Lead also determines if additional testing is required. If so, the Mobile App Lead creates a revised MoblieAppTestPlan for redistribution to the Testers and another round of testing. If testing is complete (either passed or failed) the Mobile App Lead notifies the Chain of Command, and includes the IssueLog. A test is considered a failure if the mobile application requires additional configuration to resolve MobileAppIssues. The Chain of Command notifies the Vendor of the test results. If a failure the process returns to preparing the mobile application update. If testing was completed for the test instance of the mobile application no MobileAppAcceptanceCertificate is required, and the process returns to preparing the mobile application updates, but for the production version of the application. If testing was completed for the production application and passed the Vendor sends a MobileAppAcceptanceCertificate to the Chain of Command for a signature. The Chain of Command signs the MobileAppAcceptanceCertificate and returns to the Vendor. Some MobileAppIssues may not result in a test failure, but requires a final update from the Vendor to correct; these issues are resolved through the same mobile app preparation process, but occur on an accelerated timeline.

Activity Diagram

Req#Type/Interface - Requirement DescriptionPerformance
001 Mobile App Lead/Distribute Test Plan to Testers
When the Mobile App Lead receives The Mobile App Lead distributes the updated MobileAppTestPlans to the testers. Testers are varied and include members of the CSR team, Client Relations & Information Services, Client Departments, Developers, etc.
002 Tester/Test Mobile Application Changes and Send Results to Mobile App Lead
The Testers complete their MobileAppTestPlans. If a Tester encounters a MobileAppIssue the Tester sends the MobileAppIssue to the Mobile App Lead. Testers return their tested MobileAppTestPlan to the Mobile App Lead.
003 Mobile App Lead/Collect Test Results, Update Issues Log and Test Plans, and Notify Stakeholders of Testing Progress
The Mobile App Lead collects the tested MobileAppTestPlan, and MobileAppIssues. The Mobile App Lead updates the IssuesLog, and determines if additional testing is required. If so, the Mobile App Lead revises the MobileAppTestPlan. If additional testing isn't required the Mobile App Lead Notifies the Chain of Command testing is completed. If IssueLog requires Vendor changes the Mobile App Lead sends IssueLog to Chain of Command. The Mobile App Lead sends Notification to stakeholders of testing progress.
004 Chain of Command/Notify Vendor of Testing Result
The Chain of Command notifies the Vendor of the test results. If a failure the process returns to preparing the mobile application update. If testing was completed for the test instance of the mobile application no MobileAppAcceptanceCertificate is required, and the process returns to preparing the mobile application updates, but for the production version of the application.
005 Vendor/Send Acceptance Certificate to Austin 311
The Vendor sends a MobileAppAcceptanceCertificate to the Chain of Command for a signature.
006 Chain of Command/Sign Acceptance Certificate and Return to Vendor
The Chain of Command signs the MobileAppAcceptanceCertificate and returns to the Vendor.

Austin Water Utility Mobility Pilot

Mobile Interface Realization: Working closely with our field Technicians, Austin Water Utility (AWU) identified interfaces relevant to a mobile device pilot project. The diagrams below (select image for larger, more detailed view) identify the system interface mobile function specification and is further defined in the table Austin Water Utility Mobility Interface Description below. Mobile platform includes any mobile device such as laptop, smart phone, pad device, etc. Mobile device ruggedness depends on cost and other operating environment considerations. Where appropriate, an enterprise service bus (ESB) solution should be proposed to interface with relevant water utility applications such as Hansen, Infor and Geographical Information System (GIS), etc. Currently, the City of Austin uses an IBM WebSphere Process Server ESB; however, we're exploring other ESB alternatives.

The following descriptions, diagrams and tables identify the relevant interfaces in scope of the AWU Mobility Pilot. They are identified in two categories, Water Treatment Plant and Water Distribution. Water Treatment Plant mobile pilot primarily involve Work Order management and interfaces with AWU's enterprise asset management (EAM) system Infor and operations maintenance and management system (OMMS). Water Distribution primarily involves Service Request processing using AWU's asset management system Hansen. The following paragraphs define the interface requirements in more detail.

Water Treatment Plant

In the process of correcting an identified water utility system anomaly, there are three processes important to Water Treatment Plant mobility. They include Create Work Order, Respond to Anomaly and Complete Work Order. The sequence diagrams identify the mobile interfaces required. The Interface Realization diagram provides a summary of the required mobile interfaces for this pilot and described in the table Austin Water Utility Mobility Interface Description. Their corresponding operational context can be found on the sequence diagram.

Pilot Use Case Scope

*Note: See associated use case sequence diagram for each scenario topic.

Create Work Order: When the Technician identifies a water utility system anomaly, the Technician creates a Work Order relevant to the Asset. Currently, when the Technician is unable to create a Work Order, a Work Request is generated manually and sent to the Supervisor to create the Work Order using the Enterprise Asset Management (EAM) system Infor. For mobility, the Technician needs mobility to create a Work Request on the EAM.

Respond to Anomaly: When there's a high priority response required, the Enterprise Asset Management (EAM) system notifies the Technician. When the Technician identifies an anomaly, the Technician creates a Work Order.

Complete Work Order: When a Supervisor assigns a Technician a Work Order, the Technician may identify any relevant job aids, research part availability, or complete a part Requisition. On completion, the Technician updates the Work Order status for Supervisor close out.

Complete Priority One Service Request: Based on a Customer issue, the Call Taker identifies Location History to determine if the request is a repeat call or first time call. Routinely, the Dispatcher identifies priority 1 Service Requests and provides the Service Request notification to a Technician in the vicinity of the location. Technicians use mobile device to identify the details of the Service Request and navigation route to the service location. Technicians use Geographical Information System and Hansen enterprise asset management (EAM) system to identify Water Utility Infrastructure to complete the assigned Service Request. The current workflow requires Technicians to notify the Dispatcher who in return documents the Assessment Results in Hansen. Alternately, the Technician uses mobile device to document Assessment Results.

Complete Priority Three Service Request: For Service Requests that come through the Customer Care & Billing system, the Dispatcher identifies unassigned Service Requests and assigns to Supervisors based on meter size. The Supervisor uses Hansen to schedule Service Requests. Routinely, the Technician uses Hansen to identify scheduled Service Requests and accesses information such as issue and location. Using the Geographical Information System (GIS), the Technician identifies Water Utility Infrastructure displayed on a map. Once actual location of infrastructure such as meters are identified, the Technician has an option to update the GIS location information to include photographs (if required) to assist future Technicians in meter location identification. If required, the Technician notifies the Dispatcher to create a Work Order. When a meter requires replacement, the Technician updates the meter values in the system (currently through Dispatcher). The current workflow requires Technicians to notify the Dispatcher who in return documents the Assessment Results in Hansen. Alternately, the Technician uses mobile device to document Assessment Results.

Interface Descriptions

The following interface descriptions describe the Mobile Function Specification of the relevant interface requiring a mobile solution. The related interface function can be found in context of operational use by examining the related sequence diagram. The data objects involved with the interface are identified on the Interface Realization diagram and the sequence diagrams demonstrate the instance of their use.

Each interface identifies an important object input, output and return by the function (note: output and return objects are similar).

Austin Water Utility Mobility Interface Description

Type/Interface (Function)

Description

Hansen Mobility Function Specification/
identifyServiceRequestAssignment

The field Technician shall be able to identify Service Request assignment based on a current Job Schedule.

Hansen Mobility Function Specification/
identifyServiceRequestInformation

The field Technician shall be able to identify the Issue and Location of any given Service Request along with other pertinent information. The Service Request information shall be retrieved from the Service Request Job Schedule providing a link to the Service Request information on Hansen. The Location object shall provide a link to the Navigational Route application.

Hansen Mobility Function Specification/
documentAssessmentResults

On completion of a Service Request, the Technician shall be able to document work accomplished, hours expended, parts used, etc. and close Service Request if applicable.

Hansen Mobility Function Specification/
updateMeterInformation

When a new meter is installed, the Technician shall be able to enter the new meter values into Hansen.

GIS Mobile Function Specification/
identifyInfrastructureInformation

The field Technician shall be able to identify Water Utility Infrastructure information identified by Geographic Information System on a Location Map linked from a particular Location object/attribute from the Service Request.

GIS Mobile Function Specification/
updateGISInformation

The field Technician shall be able to input updated GIS Information regarding Water Utility Infrastructure to correct incorrect location information or other relevant and modifiable GIS attributes.

Dispatch Mobile Function Specification/
prepareWorkOrder

The field Technician shall be able to prepare a Work Order based on a current Service Request and the Service Request Status. Currently, the Dispatcher at the request of the field Technician is accomplishing this activity.

Dispatch Mobile Function Specification/
documentNewMeterValues

The field Technician shall be able to establish new meter readings when required - such as when a meter is replaced. Currently, the new meter value is communicated to the Dispatcher who enters the new meter values.

Dispatch Mobile Function Specification/
documentAssessmentResults

The field Technician shall be able to document assessment results based on open Service Request. Currently, this information is communicated through the Dispatcher who enters the assessment information into Hansen.

Dispatch Mobile Function Specification/
obtainOneCallAndExcavationPermit

Currently, the field Technician notifies the Dispatcher who in return prepares a Work Order using Hansen. Using a mobile device, the field Technician shall be able to submit for One Call and Excavation Permits. Objects/attributes contained in the Service Request pertinent to the to One Call and Excavation Permit shall be automatically transferred to the One Call and Excavation Permit submission.

Navigation Mobile Function Specification/
identifyNavigationalRoute

The field Technician shall be able to use Service Request Location information to navigate to the service site.

Technician Mobile Function Specification/
serviceTheRequest

The Dispatcher shall be able to communicate Notification containing Service Request identification to field Technician. An Acknowledgment shall be returned to the Dispatcher indicating receipt of Notification. The Notification shall contain a link to identify Service Request information to include service location on Hansen.

Technician Mobile Function Specification/
assessWorkToPerform

Supervisor shall be able to communicate high priority Work Orders to the field Technician. An Acknowledgment shall be returned to the Supervisor indicating field Technician receipt. The Notification shall contain a link to identify Work Order information located on Enterprise Asset Management system.

Technician Mobile Function Specification/
createWorkRequest

The field Technician shall be able to create a Work Request for identified Treatment Plant anomalies.

OMMS Mobile Function Specification/
identifyJobAids

The Technician shall be able to access job aids on Operations Maintenance and Management System (OMMS) and Microsoft SharePoint. Allows Technician to find procedures, instructions and other job aids.

EAM Mobile Function Specification/
researchPartAvailability

The Technician shall be able to perform research for parts availability to include part location.

EAM Mobile Function Specification/
completeRequisition

The Technician shall be able to complete a part requisition.

EAM Mobile Function Specification/
completeWorkOrder

The field Technician shall be able to close out a work order by modifying current work order status.

EAM Mobile Function Specification/
createWorkOrder

The field Technician shall be able to create a Work Order to include status of any given Asset.

EAM Mobile Function Specification/
enterMeterReading

The field Technician shall be able to enter Meter Reading based on scanned bar codes identifying the Asset.

EAM Mobile Function Specification/
identifyWorkOrderInformation

The field Technician shall be able to select link (Work Order Identification) from Supervisor Notification to retrieve the Work Order from the Enterprise Asset Management system.

Asset Mobile Function Specification/
scanBarCode

The mobile device shall be able to identify the Asset based on bar codes affixed to the Asset.

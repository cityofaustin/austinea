Process Payment
Use Case Model

Scope: Austin Code Finance receives a payment either by mail or from an Austin Code Program Specialist. If the payment is received through mail, Austin Code Finance processes the payment by opening the mail in pairs with the second staff member acting as an auditor. If payment is received from the Austin Code Program Specialist, a receipt is returned to the Austin Code Program Specialist. Austin Code Finance updates the FinanceTrackingSheet with details about the payment. If a receivable exists for the payment, Austin Code Finance uses AIMS to update the receivable. The payment is then prepared for deposit. For Licensing Payments, the Austin Code Program Specialist prepares the deposit partially and then delivers it to Austin Code Finance to complete the deposit. Austin Code Finance updates the FinanceTrackingSheet, fills the DepositBag, updates the DepositBook, and creates a DepositTicket. Deposits are picked up throughout the business week by a security firm. After the deposit has been picked up, Austin Code Finance uses the BankSearch system to check that the deposit was successfully made to the bank. Once the deposit is verified in BankSearch, Austin Code Finance prints the BankSearch verification. Austin Code Finance uses AIMS to create a CashReceipt, and then assembles a CashReceiptPacket containing the CashReceipt, BankSearch, copies of checks, DepositTicket, etc. Austin Code Finance sends the CashReceiptPacket to the Chain of Command for approval. The Chain of Command approves the CashReceipt in AIMS. If the payment needs to be recorded in the Case Management System, Austin Code Finance updates the appropriate CodeViolationCase with payment details.

Process Payment
Sequence Diagram

Req#Type/Interface - Requirement DescriptionPerformance
001 Austin Code Finance/processPaymentByMail
An Austin Code Finance staff opens the received mail while another staff member observes the opening for auditing purposes (may include Austin Code Program Specialist). Austin Code Finance logs the payment type into the FinanceTrackingSheet.
002 Austin Code Finance/processPayment
The Austin Code Program Specialist requests Austin Code Finance to process a payment. Austin Code Finance returns a receipt to the Austin Code Program Specialist and then updates the FinanceTrackingSheet.
003 AIMS/updateReceivable
Austin Code Finance uses the system to update the receivable by adjusting amount owed, accounting for interest, based on time elapsed, Chain-of-Command adjustment, etc.
004 Austin Code Finance/prepareDeposit
Austin Code Finance updates the FinanceTrackingSpreadsheet (from, check number, type of payment, amount of payment, payment for [lien, short-term rental and type], fund, department, unit, revenue code [always starts with a 4], and comments). Austin Code Finance creates a DepositTicket (one per day). Liens receive their own DepositTicket. DepositTicket contains: currency amount, coin amount, checks (multiple lines and listed individually, check amount, check number), total for all deposits, and deposit ticket number. Austin Code Finance prepares a DepositBag and the Dunbar Booklet is updated: description (majority checks or cash), total amount of deposit, bag number (not filled out). The Dunbar Booklet Number sticker (barcode number for armored services) is attached to the DepositBag. Austin Code Finance prepares the deposit for pick up from Dunbar Armored Services. Pick up occurs Mondays, Wednesdays, and Fridays. Dunbar Armored Services fills out the Dunbar Booklet (sign, date, time).
005 Austin Code Program Specialist/prepareDeposit
Licensing only receives payments for walk-ins, nothing by mail. Austin Code Program Specialist logs the deposit (from, date received, receipt number, address, check number, amount, type of license [comment field short-term rental, waste hauler, etc.], funding source [fund, department, unit, revenue code], total deposit, deposit ticket number, deposit when created, armored service pick up, barcode number for armored services, and cash receipt from AIMS). Austin Code Program Specialist prepares the DepositTicket, the DepositBag, and assembles portions of the CashReceiptPacket (everything minus BankSearch). The Austin Code Program Specialist then delivers the DepositBag to Austin Code Finance and the Dunbar Booklet is updated (description [majority checks/cash], total amount of deposit, bag number [not filled out]). The Dunbar Booklet Number sticker (barcode number for armored services) is attached to the DepositBag.
006 Bank Search/verifyBankDeposit
Austin Code Finance uses the FinanceTrackingSheet to verify the bank deposit using the system. The system is checked daily. Once a deposit is verified, the BankSearch verification is printed.
007 AIMS/createCashReceipt
The user creates the CashReceipt: fund, department, unit, revenue code (always starts with a 4), record date, deposit date, deposit ticket, document description, extended description (what CR is for, type of license, etc.), bank account number, legal name (what payment was for, type of licensing), payment date, check EFT number, AR department (1600), AR unit (all), event type (type of deposit creating), line description (varies), dollar amount, fund, department, unit, revenue code, and receivable (RENumber) if present. The user then submits the CashReceipt into the system.
008 Austin Code Finance/assembleCashReceiptPacket
Austin Code Finance assembles a CashReceiptPacket. Packet includes printouts of CashReceipt, CashLog/CashReport, BankSearch printout, DepositTicket, copies of checks (with redacted check number, bank account number and accounting number), and receivable if present.
009 Chain of Command/approveCashReceiptPacket
Austin Code Finance sends the CashReceiptPacket to the Finance Chain of Command for approval. All documents received each month must be approved by month's end. The Chain of Command approves the CashReceipt in the system, initials the CashReceiptPacket, and returns it to Austin Code Finance for filing.
010 AIMS/approveCashReceipt
The user uses the system to approve the CashReceipt, which is based on the CRNumber.
011 Case Management System/updateCodeViolationCase
The user updates the CodeViolationCase with relevant information to the case. If the CodeViolationCase generated a StopWorkOrder, the StopWorkOrder is attached to the CodeViolationCase. The user attaches a copy of the DiagramOfWorkRequired. If a board and secure occurs, a BoardAndSecureBill is attached to the CodeViolationCase by Austin Code Admin. If a sign was removed, the Austin Code Officer adds the sign and information to the CodeViolationCase. Additionally, if CautionBarriers were placed, information on the CautionBarriers is added to the CodeViolationCase. If property information was updated, the Case Management System includes updated property information. The user updates the CodeViolationCase with information relevant to the change. This includes, but is not limited to, BoardAndSecureBill, PropertyAbatementBill, FollowUpNotes, photographs (boarded, non-boarded, posted notifications, placards), emails, PhoneCallSummary, other relevant information and inter-department communications. If a ServiceCase is created, the Public Information Office Admin adds notes to the CodeViolationCase including the CitizenNeed and ServicesRecommended for ServiceCase. If CorrectiveResources are identified, the user updates the CodeViolationCase with a list of CorrectiveResources provided (three resources) and notes of contact with Owner/Tenant. HardshipReason is added to the CodeViolationCase. If new violations are identified, the user adds the new violation to the Case Management System and the violation is attached to the original CodeComplaintCase. If a TitleSearch is completed, the TitleSearch is added to the CodeViolationCase. If additional LegallyInterestedParties are identified, the parties are added to the CodeViolationCase. If a NoticeOfViolation is generated, the NoticeOfViolation is added to the CodeViolationCase. If posted at the property, photographs are attached to the CodeViolationCase. If the NoticeOfViolation is delivered and a certified mail receipt is received, the receipt is added to the CodeViolationCase. If a citation is created, the Case Management System attaches the citation to the CodeViolationCase. Additionally, the user scans and uploads a copy of the physical citation to the Case Management System. If an AdministrativeHearingWarning is issued, the user updates the CodeViolationCase with information on the issued AdministrativeHearingWarning. If an AdministrativeHearingCitation is issued, the user updates the CodeViolationCase with the AdministrativeHearingCitation, and a LegalCase is created. The user scans and uploads returned liens to the CodeViolationCase. If a lien is filed for the property, the TRVNumber is added to the CodeViolationCase. The Austin Code Officer updates the CodeViolationCase with information concerning Violation extensions, written request, reason for approval/denial, etc. If there is a payment then the CodeViolationCase is updated by the user to include the payment.
